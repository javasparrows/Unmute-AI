# テスト戦略

## なぜ文マッピング精度のバグが繰り返し漏れるのか（反省）

### 過去の失敗パターン

1. **短い簡潔なテストテキストだけで通過判定してしまう**
   - 3文程度の短文テストでは、略語のピリオド (`e.g.`, `et al.`) が含まれず、文数一致のまま pass する
   - 実際のアカデミックテキストには必ず略語が含まれ、偽の文境界が発生する

2. **左→右方向だけ確認して逆方向をスキップしてしまう**
   - 左→右は「たまたま合っている」ことが多い（左パネルの文数が基準になるため）
   - 問題は常に **右→左（翻訳→原文）方向** で発生する

3. **目視確認のみで、文数の一致を数値的に検証していない**
   - 「なんとなくハイライトが当たっている」で pass にしてしまう
   - 特に長文では 1 文ズレを目視で見落としやすい

4. **テキスト変更後に即テストせず、build pass のみで完了としてしまう**
   - ハイライト同期は TypeScript の型チェックでは検出できない
   - **必ずブラウザでの手動検証が必要**

### 対策: 今後のテスト必須プロトコル

以下を **コード変更のたびに** 実施する。

## テスト観点

### 1. 文マッピング精度（最重要 — 漏れやすい）

**プログラム検証（DevTools Console）:**
```javascript
// 両パネルの文数を比較するスクリプト
const editors = document.querySelectorAll('.tiptap');
const leftText = editors[0]?.textContent || '';
const rightText = editors[1]?.textContent || '';
const REGEX = /([.。!?！？])/g;
function split(text) {
  return text.split('\n\n').flatMap((p, i) => {
    const r = []; let last = 0; REGEX.lastIndex = 0; let m;
    while ((m = REGEX.exec(p)) !== null) {
      r.push(p.slice(last, m.index + 1)); last = m.index + 1;
    }
    if (last < p.length) r.push(p.slice(last));
    return r.filter(s => s.trim());
  });
}
const L = split(leftText), R = split(rightText);
console.log(`Left: ${L.length}文, Right: ${R.length}文, 差: ${L.length - R.length}`);
if (L.length !== R.length) console.warn('⚠️ 文数不一致 → ハイライトにズレが発生する');
```

**目視検証（以下の全文でクリック確認）:**
- 必ず **右パネル→左パネル方向** を先にテストする（ズレが出やすい方向）
- 段落1の最初の文、中間の文、最後の文
- 段落2の最初の文、最後の文
- 特に `e.g.`、`et al.`、`i.e.` を含む文の前後

### 2. ハイライト同期（双方向）

- 原文クリック → 翻訳のハイライトが意味的に対応する文を指していること
- 翻訳クリック → 原文のハイライトが意味的に対応する文を指していること
- ハイライトが文字の高さと合っている
- 段落をまたいだ場合でもズレないこと

### 3. テキスト入力・永続化

- 文章の paste ができる
- 同期ボタン押下で翻訳が実行される（自動翻訳は発生しないこと）
- リロード後もテキストが保持される（localStorage）
- 言語設定・ジャーナル設定・プロバイダ設定もリロード後に保持される

### 4. 差分翻訳

- 1文変更→同期で、変更文のみが翻訳更新されること
- 未変更文が前回の翻訳のまま保持されること
- 段落追加/削除が正しく処理されること

### 5. エラー耐性

- クリア後に文数が異なるテキストを入力してもエラーが発生しない
- API エラー時にエラーメッセージが表示され、クラッシュしない
- アラインメントAPI失敗時に位置ベース1:1フォールバックが動作すること

---

## テストデータ要件

- **必ず `test_cases.md` のアカデミックテキストを使用する**（`e.g.`、`\cite{...}`、`---` を含む）
- 短い 3 文テストだけで pass にしない
- テストテキストに以下を必ず含めること:
  - 略語 (`e.g.`, `et al.`, `i.e.`, `Fig.`)
  - 長いセミコロン付き複文（`;`の後にピリオドなしで続く文）
  - 2段落以上
  - LaTeX記法（`\cite{...}`）

---

## テスト実行のタイミング

| 変更内容 | 必須テスト |
|---|---|
| ハイライト関連のコード変更 | 文マッピング精度（全項目） |
| 翻訳フロー/同期hookの変更 | 差分翻訳 + 文マッピング精度 |
| UIレイアウト変更 | ハイライト目視 |
| 新機能追加 | 全項目 |
| build pass のみ | **不十分** — 必ずブラウザ検証を追加 |
